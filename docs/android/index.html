This is the cert whose SHA-256 the on-chain verifier requires. Change it and your proof won't verify against the pinned root.

    


    

        The on-chain verifier accepts only chains terminating in this specific cert. To accept additional Google roots (as they rotate), the contract owner calls addRoot(sha256) — that's the only governance lever.
    





What you can check yourself



    

        
Policy lives in allowedRootFingerprints

        

            Owner is 0x5A37…AD8D; only they can call addRoot / removeRoot. Query any pin yourself via
            BaseScan Read Contract
            → allowedRootFingerprints(bytes32) → bool.
            The pinned-root pill above runs that exact call every 12 s.
        

    

    

        
verify(bytes) is view

        

            Marked view in
            AndroidKeyAttestationVerifier.sol.
            Same code path costs ~50M gas if called as a state-changing tx (measured via cast estimate) — over Base's 16.7M per-tx cap. As a view call it has no cap and is deterministic: same proof bytes → same (codeId, pubkey, challenge) tuple every time, for everyone.
        

    

    

        
Pubkey is bound to the challenge pattern

        

            Each attestation binds the leaf's EC pubkey to a caller-supplied challenge inside the signed StrongBox cert. Two attested parties can use that to ECDH off-chain without an on-chain handshake. Not yet demoed end-to-end here — only the verifier side is live; the pairing flow is the next demo.
        

    





Re-verify an attestation



    
    
    
    
    

        
        
    

    

        Re-verification is a stateless eth_call against the Base Sepolia verifier. The proof bytes never leave your browser before being submitted to the public RPC. The verifier returns (codeId, leafDer, challenge) iff the chain signatures + root pin + policy all pass.

What's actually happening



    Each chain link is verified inside the EVM:
    
    
        
  • leaf → cert1 : ECDSA-P256-SHA256 (via vendored daimo P256Verifier)
    • 
        
  • cert1 → cert2 : ECDSA-P384-SHA256 (ported from estid-sig P-384 / Estonian e-ID)
    • 
        
  • cert2 → root : RSA-PKCS1v15-SHA256 (via modexp precompile)
    • 
        
  • root : SHA-256 pin in allowedRootFingerprints
    • 
    

    Plus on-chain DER parsing of the leaf to recover (challenge, deviceLocked, verifiedBootState) from the Android Key Attestation extension at OID 1.3.6.1.4.1.11129.2.1.17. Source: teleport-computer/tee-interop.
