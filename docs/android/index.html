Pixel as a publicly verifiable, permissionless TEE

A Pixel 6's Titan M2 is a single-die integrated secure element with attestations rooted in a Google-published hardware key. Combined with APK sideloading via adb install — no Play Store, no developer account, no infrastructure between you and your phone — any Pixel can act as an interoperable peer in a publicly verifiable TEE network. Below: live evidence of a Pixel and a GitHub-Actions runner exchanging a secret end-to-end, both sides independently verifiable.

What makes this work

Integrated TEE. Titan M2 in the Pixel 6 puts the attestation signing key and the secure-boot ROM on the same silicon die. Bus-master substitution — the attack that breaks discrete-TPM-on-Pi or STSAFE-on-MCU setups — doesn't work against it. Same trust model as Intel SGX, AMD SEV-SNP, or AWS Nitro Enclave, on consumer hardware around $300.

Publicly verifiable. The chain terminates in Google's published Hardware Attestation Root — the same root every other verifier in the ecosystem uses. Re-verification is a stateless eth_call against a deployed Solidity verifier; the pairing artifact is also Sigstore-witnessed so the GitHub-Actions side is independently re-verifiable via cosign verify-blob. Two unrelated public CAs sign off on one end-to-end claim.

Permissionless. Attestation binds to hardware, not to app distribution. adb install works on any consumer Pixel once USB debugging is enabled — no Google Play developer account, no app review, no signing key from us. You write the code, install it on your own phone, and the StrongBox attests to whatever key it generated. See how sideloading composes with attestation below.

What just happened. The GHA workflow above:
  1. Read the Pixel 6 attestation chain from the repo.
  2. Verified it locally — same chain math as the on-chain verifier (Google root → cert2 → cert1 → leaf).
  3. Extracted the leaf's P-256 SubjectPublicKey (shown above as attested SPKI).
  4. Generated an ephemeral P-256 keypair on the runner. The private half dies when the runner tears down.
  5. Did ECDH: shared = ephemeral_priv × attested_pub. HKDF-SHA256 → AES-256-GCM key.
  6. Encrypted a fresh 32-byte random secret. Committed the bundle. Committed the SHA-256 of the plaintext as a public commitment.
  7. Cosign-signed the bundle, anchoring its origin in this workflow at this commit.
Only the Pixel device holding the StrongBox-resident private key can recover the plaintext, by computing the same ECDH from its side. The committed plaintext_sha256 lets the recovery be publicly confirmed.

Why this is permissionless

The APK that produced the chain above lives in this repo at pixel-attest/. Two activities (~280 lines of Java), built with javac → d8 → aapt2 → apksigner. No Gradle, no Android Studio. Signed APK is ~16 KB.

To run on your own Pixel — no Play Store interaction needed:

# once: enable Developer Options → USB debugging, accept the RSA prompt
adb install edgetee-attest.apk
adb shell am start -n com.edgetee.attest/.MainActivity \
    -e challenge "your-challenge" \
    -e alias "your-alias" \
    -e strongbox true
adb pull /sdcard/Android/data/com.edgetee.attest/files/attestation.pem

Titan M2 doesn't care which app made the keygen request. It records the caller's signing-cert digest in the attestationApplicationId field (a fact recorded inside the chain, not a permission check) and proceeds. Our on-chain verifier doesn't constrain that field either — any sideloaded APK signed with any debug keystore can produce a valid chain. The cryptographic chain still terminates at Google's hardware root.

This is the layer-distinction from iOS App Attest or Play Integrity API: those bind the attestation to an app distributed via the platform store. Android Key Attestation operates one layer lower — it attests to the device's secure-element-held key, regardless of app provenance. The result is the same hardware-trust guarantee with no platform-distribution gatekeeper.

How it composes with other TEEs

tee-interop is a multi-verifier registry: each attestation platform plugs in as an ERC-733 IVerifier. The registry has zero platform-specific code; the demo above pairs Android Key Attestation with Sigstore via GHA OIDC, but the same composition works between any two registered verifiers (Android + Nitro, SGX + Pixel, etc.).

Each chain link in the Pixel side is verified inside the EVM:

  • leaf → cert1 : ECDSA-P256-SHA256 (via vendored daimo P256Verifier)
  • cert1 → cert2 : ECDSA-P384-SHA256 (ported from estid-sig Estonian e-ID P-384)
  • cert2 → root : RSA-PKCS1v15-SHA256 (via the modexp precompile, adria0/SolRsaVerify)
  • root : SHA-256 pin in allowedRootFingerprints

Plus on-chain DER parsing of the leaf to recover (challenge, deviceLocked, verifiedBootState) from the Android Key Attestation extension at OID 1.3.6.1.4.1.11129.2.1.17. verify(bytes) is marked view in AndroidKeyAttestationVerifier.sol — the same code path would cost ~50M gas as a state-changing tx (over Base's 16.7M per-tx cap) but is unconstrained as a view call. Same proof bytes → same (codeId, pubkey, challenge) tuple every time, for everyone.
