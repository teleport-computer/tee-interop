Pixel Key Attestation is not a code-attesting TEERETRACTED

This page used to claim "Pixel as a publicly verifiable, permissionless TEE" and demonstrated a Pixel ↔ GitHub-Actions secret exchange "both sides independently verifiable." The interactive demo has been taken down because that framing was wrong. The full write-up is in README.md.
The short version. Android Key Attestation never measures the app's code — it binds the app's signer (attestationApplicationId = package + signing-cert digest), not what the code does.
  • Permissionless sideloading ⇒ no canonical signer; anyone signs their own APK and gets a valid Google-rooted chain. The signer constrains nothing.
  • An evil twin (same package, same signer) that exfiltrates the challenge produces a byte-identical attestation that passes the fullest pin set.
  • Non-extractability protects the key bytes, not the data: keystore access is gated by app identity, not code, so any code under that identity can use the key to decrypt and exfiltrate. "Only this device can decrypt" carries no confidentiality.
  • Not the SGX/SEV/Nitro trust model — those measure the enclave image; this measures only the OS (verified boot), never the app.
What a Pixel attestation genuinely proves: genuine Titan M2, locked bootloader + verified boot, a non-extractable key, and a chain to Google's Hardware Attestation Root. That is proof of genuine, unique secure hardware (good for sybil resistance) — ERC-733 Stage 0 for code integrity, not a TEE that runs attested code.
Demonstration + repro: edge-tee/pixel-attest/NOTES-attestation-does-not-bind-code.md. The on-chain verifier (contracts/AndroidKeyAttestationVerifier.sol) remains a correct cert-chain verifier; it just does not verify any code.
