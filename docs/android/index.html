Architecture

Policy on-chain
Pinned root set is governance state. Owner adds / removes Google attestation roots via cheap txs.
Verification view-only
verify(proof) runs the full ECDSA-P256 + ECDSA-P384 + RSA-PKCS1v15 + DER chain inside the EVM. ~50M gas — but it's a view call, so no gas cap.
Secret exchange off-chain
Two attested parties verify each other's proofs as view calls, then ECDH using the attested pubkeys. No on-chain state writes needed for pairing.

Re-verify an attestation

Re-verification is a stateless eth_call against the Base Sepolia verifier. The proof bytes never leave your browser before being submitted to the public RPC. The verifier returns (codeId, leafDer, challenge) iff the chain signatures + root pin + policy all pass.

codeId = keccak256 of the leaf cert. Stable per device per attestation. pubkey = leaf cert DER (~700 bytes; SPKI inside binds the per-attestation EC pubkey). challenge = the bytes the StrongBox signed over — bind your own session keys here for pairing.

What's actually happening

Each chain link is verified inside the EVM:
  • leaf → cert1 : ECDSA-P256-SHA256 (via vendored daimo P256Verifier)
  • cert1 → cert2 : ECDSA-P384-SHA256 (ported from estid-sig P-384 / Estonian e-ID)
  • cert2 → root : RSA-PKCS1v15-SHA256 (via modexp precompile)
  • root : SHA-256 pin in allowedRootFingerprints
Plus on-chain DER parsing of the leaf to recover (challenge, deviceLocked, verifiedBootState) from the Android Key Attestation extension at OID 1.3.6.1.4.1.11129.2.1.17. Source: teleport-computer/tee-interop.
